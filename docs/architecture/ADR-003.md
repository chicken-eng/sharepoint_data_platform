# ADR-003: Multi-Layer PostgreSQL Database Schema Architecture

| Metric | Value |
| :--- | :--- |
| **Status** | Approved |
| **Date** | July 2026 |
| **Deciders** | Lead Data Engineer / Database Administrator |

## Context & Problem Statement
Historical files across SharePoint contain inconsistent column names, varied date formats, missing fields, and unvalidated data. Inserting this raw data directly into production operational tables will break constraints and make historical analysis unreliable.

## Decision Drivers
* Need for auditability and raw data retention.
* Ability to clean and transform data incrementally without re-downloading files.
* Sub-second query performance for respondent email lookups.

## Decision Outcome
We adopt a **5-Schema Layered Architecture** inside PostgreSQL:

1. **`metadata`**: Pipeline catalog tracking folder structures, file metadata, hashes, and sync statuses.
2. **`bronze`**: Raw tabular representations (using `JSONB` or loose string columns) preserving uncleaned source values.
3. **`silver`**: Cleaned, strongly typed, deduplicated relational entities mapped to canonical domain models (`respondents`, `projects`, `clients`).
4. **`gold`**: Search-optimized indexes (e.g., email lookup tables with B-tree and trigram indexes) and aggregate views for analytics.
5. **`audit`**: Pipeline execution logs, run stats, row counts, and operational alerts.

### Positives & Benefits
* **Clear Separation of Concerns:** Raw data, cleaned data, and business-ready search indexes never mix.
* **Debugging Efficiency:** If an email is improperly formatted in `silver`, developers can inspect the exact raw string in `bronze` and the original file in `metadata` to debug the parsing rule.
* **Performance Optimization:** Operational search indexes in `gold` remain lean, fast, and optimized without carrying unused metadata columns.
