# ADR-001: Adoption of MinIO Object Storage as Raw Landing Zone

| Metric | Value |
| :--- | :--- |
| **Status** | Approved |
| **Date** | July 2026 |
| **Deciders** | Lead Data Engineer / Database Administrator |

## Context & Problem Statement
The historical data pipeline must pull over 100 GB of multi-format files (`.xlsx`, `.csv`, `.docx`) from two Microsoft SharePoint environments. Directly parsing these files from Microsoft Graph API streams into PostgreSQL creates significant architectural vulnerabilities:
1. Microsoft Graph API applies aggressive rate-limiting (`HTTP 429 Too Many Requests`).
2. If parsing code fails halfway through a job, the system must re-fetch files over the internet, risking API blocking and extended execution times.
3. SharePoint files can be altered or deleted upstream, leading to non-reproducible pipeline executions.

## Decision Drivers
* Zero licensing/cloud infrastructure cost.
* Need for strict decoupling between I/O (downloading) and Compute (parsing/transformation).
* Immutability and replayability of historical data pipeline runs.

## Considered Options
1. **Direct Stream Parsing:** Stream files from Graph API directly into Python memory and insert into PostgreSQL.
2. **Local Disk Storage:** Save downloaded SharePoint files directly to the host machine filesystem.
3. **MinIO Object Storage:** Run a local S3-compatible MinIO container as a dedicated raw data lake.

## Decision Outcome
**Chosen Option:** **Option 3 (MinIO Object Storage)**.

### Positives & Benefits
* **API Shielding:** Files are fetched once from SharePoint and saved to MinIO. Subsequent re-runs or parse failures read from local MinIO at gigabit speed with zero Graph API calls.
* **S3 API Compatibility:** MinIO uses standard S3 SDKs (`boto3`). If the organization eventually migrates to AWS S3 or Azure Blob in the future, zero pipeline code needs to change.
* **Immutability:** The raw landing zone preserves the exact state of the SharePoint file at the time of extraction.

### Negatives & Trade-offs
* Requires local disk capacity (~100–150 GB) on the deployment host.
* Adds one additional service container (`minio`) to the `docker-compose.yml` stack.
