# ADR-002: Ingestion Engine Tech Stack Selection (Python + Polars/OpenPyXL)

| Metric | Value |
| :--- | :--- |
| **Status** | Approved |
| **Date** | July 2026 |
| **Deciders** | Lead Data Engineer / Database Administrator |

## Context & Problem Statement
We require a robust, lightweight processing engine to crawl SharePoint API endpoints, manage MinIO file transfers, extract multi-sheet Excel data, and execute SQL transformations in PostgreSQL.

## Decision Drivers
* Total data volume is ~100 GB (fits easily within single-node memory and CPU parameters).
* Need for high-performance Excel and CSV parsing without memory overhead.
* Developer familiarity and maintainability.
* Zero licensing cost.

## Considered Options
1. **Java/Scala + Apache Spark:** Distributed compute stack.
2. **GUI ETL Tools (e.g., Talend / Pentaho):** Drag-and-drop integration tools.
3. **Custom Python Stack (`httpx`, `openpyxl`, `polars`, `psycopg3`, `SQLAlchemy`):** Modular Python ingestion pipeline.

## Decision Outcome
**Chosen Option:** **Option 3 (Custom Python Stack)**.

### Justification
* **Overengineering Avoidance:** Apache Spark introduces massive cluster overhead (JVM, master/worker nodes) that is unjustified for a dataset under 1 TB.
* **Memory Management:** Using `polars` (Rust-backed DataFrame engine) and `openpyxl` (in `read_only=True` mode) provides memory-efficient, fast parsing on standard hardware.
* **Testability:** Python code can easily be unit-tested using `pytest`, mocked, and integrated into standard CI/CD pipelines.

### Positives & Benefits
* Fast execution speeds with low CPU/RAM footprint.
* Complete control over retry mechanisms, backoff logic, and Graph API handling.
* Clean separation of concerns using object-oriented Python modules.

### Negatives & Trade-offs
* Column mapping and schema evolution must be handled explicitly in custom Python/SQL logic rather than built-in GUI auto-mappings.
